Intro-Lab: OAuth 2.0 Authorization Grant Flows:

In this intro lab you will use curl, httpie or postman to create the OAuth 2.0 requests to get access tokens.

A client can request an access token using only their client credentials. Table 1 shows the parameters of the client’s credentials.

Table 1 - Parameters of the client’s credentials

Parameter Value
token url http://localhost:8080/auth/realms/workshop/protocol/openid-connect/token
grant_type client_credentials
client_id library-client
client_secret 9584640c-3804-4dcd-997b-93593cfb9ea7
scope library_user email profile

curl

To get an access token using curl, I used the following command:

curl -X POST -H "Content-Type: application/x-www-form-urlencoded" -d "grant_type=client_credentials&client_id=library-client&client_secret=9584640c-3804-4dcd-997b-93593cfb9ea7" http://localhost:8080/auth/realms/workshop/protocol/openid-connect/token

This command sends a POST request to keycloak with the content type in the header = application/x-www-form-urlencoded.This means that the values are encoded in tuples of keys separated by the “&” character, with “=” between the key and the value.
This command will send the parameters grant_type, client_id, client_secret.

As a result, I get an access token using curl, as shown in Figure 4.

RO Password Credentials Grant

The resource owner password credentials grant type is suitable in cases where the resource owner has a trust relationship with the client. The authorization server should take special care when enabling this grant type and only allow it when other flows are not viable.

This type of grant is also used to migrate existing clients using direct authentication schemes, such as HTTP Basic or Digest Authentication, to OAuth by converting stored credentials into an access token.

The necessary parameters for providing RO password credentials are shown in Table 2.

Table 2 - RO password credentials

grant_type password
client_id library-client
client_secret 9584640c-3804-4dcd-997b-93593cfb9ea7
scope library_user email profile
username bwayne
password wayne

Advanced: Authorization Code Grant

The authorization code grant type is used to obtain both access tokens and refresh tokens and is optimized for confidential clients. Since this is a redirection-based flow, the client must be capable of interacting with the resource owner’s user-agent (typically a web browser) and capable of receiving incoming requests (via redirection) from the authorization server.

Figure 14 - Access token (postman)

The access token is used when making requests to the server (for example, when logging in). It has two properties: it is reusable and short-lived.

The refresh token is used to update a pair of access and refresh tokens. It also has two properties opposite to the first token: it is disposable and durable.

When the access token lifetime comes to an end, the application uses refresh token to update both tokens and continue using the new access token

Thus, the refresh + access token scheme limits the time for which an attacker can access the service. Compared to one token, which an attacker can use for weeks and no one will know about it.

Right after installation, you will get two databases, or suffixes: one for your data, based on your host’s domain (dc=example,dc=com), and one for your configuration, with its root at cn=config. To change the data on each we need different credentials and access methods:

So now I have two databases: one for the data based on the host domain (dc=st12, dc=sne22, dc=ru), and one for the configuration with the cn=config root. To change the data on each, different credentials and access methods are used:

  1. dc=st12,dc=sne22,dc=com: the administrative user is cn=admin,dc=st12,dc=sne22,dc=com and its password is the one selected during the installation of the slapd package;
  2. cn=config: the configuration slapd itself is stored under this database.

The setup started with the following block of commands:

This will install a module for the PAM subsystem (Pluggable Authentication Modules) that allows me to store a single database of system information in the LDAP directory - users, groups, computers, services, etc., which can be shared by many Linux systems.

Nscd is a daemon that provides a cache for the most common name service requests.

During the installation, I need to set the configuration (Figures 37, 38, 39):

Now I need to configure several files to tell our authentication files that they can access our LDAP server for authentication information.

First, I edited the /etc/nsswitch.conf file as shown below:

passwd:         ldap compat  
group:          ldap compat  
shadow:         ldap compat

This will allow me to specify that the LDAP credentials should be modified when users issue authentication change commands.

Then I changed the /etc/pam.d/common-session file by adding the line session required pam_mkhomedir.so skel=/etc/skel umask=0022 at the end.

This will create a home directory on the client machine when an LDAP user logs in who does not have a home directory.

To make changes, I restarted the service using the command sudo /etc/init.d/nscd restart.
